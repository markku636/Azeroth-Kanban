## 目標

實作 Kanban 看板的核心功能：

1. **CRUD API** — `/api/v1/kanban/cards/*` 提供卡片新增 / 列表 / 編輯 / 刪除；ownerId 由 session 取得，不接受 client 傳入
2. **Reorder API** — `/api/v1/kanban/cards/[id]/move` 處理拖拉後的狀態 + sortOrder 更新；採整數中位數演算法，必要時觸發欄內 normalize
3. **Kanban 頁** — `/admin/kanban` 4 欄式版面（待處理 / 進行中 / 待驗收 / 已完成）+ 頂端 inline 新增表單 + 卡片 hover 浮現編輯 / 刪除 icon + 編輯 Modal（含 emoji 狀態下拉）
4. **dnd-kit 整合** — `@dnd-kit/core` + `@dnd-kit/sortable` + Pointer / Keyboard sensors（觸控強化由 Spec C 負責）；optimistic UI + 失敗回滾
5. **Toast 系統** — 在 `(dashboard)/layout.tsx` 掛載 `<Toaster />`，所有 CRUD / 拖拉操作回饋走 toast（成功綠 / 失敗紅）
6. **Sidebar 入口** — `menu-items.tsx` 新增 Kanban 連結（`PERMISSIONS.KANBAN_VIEW`）；`routes.ts` 補 kanban route

## 邏輯變更點

### Service 層

- **sortOrder 算法**（純函式，可單元測試）：
  - 新增卡片落「待處理」末端：`MAX(sort_order WHERE owner=u AND status=TODO) + 1000`，無同欄卡片時 `1000`
  - 拖拉至兩卡片之間：`(prev.sort_order + next.sort_order) / 2`
  - 拖拉至首位：`next.sort_order - 1000`，若 `< 1` 觸發 normalize
  - 拖拉至末位：`prev.sort_order + 1000`
- **Normalize**：當欄內某兩相鄰卡片 sortOrder 差距 `< 2`，下一次插入無中位整數 → 觸發批次 UPDATE 將該欄 sortOrder 重排為 1000, 2000, 3000…
- **Owner 過濾**：所有 query 一律 `where: { ownerId: session.user.memberId }`，admin 也不例外（PRD § 5 跨使用者隔離）

### API 層

- API route 只負責驗證 + 轉發 service 結果，不含業務邏輯
- 全部用 `withPermission(PERMISSIONS.KANBAN_*)` 守衛
- Body 驗證：用 Zod schema，違規回 `ApiReturnCode.VALIDATION_ERROR` + `ApiErrorCode.KANBAN.TITLE_REQUIRED` / `TITLE_TOO_LONG` 等

### 頁面與互動

- Kanban page 使用 `useState<KanbanBoardState>` 結構：`{ TODO: Card[], IN_PROGRESS: Card[], IN_REVIEW: Card[], DONE: Card[] }`
- `useKanbanBoard` hook：
  - `loadBoard()`：fetch `/api/v1/kanban/cards` → 分欄 state
  - `addCard(title, description?)`：optimistic 加到 TODO 末端 → POST → 失敗回滾 + toast
  - `updateCard(id, patch)`：optimistic 更新 → PATCH → 失敗回滾
  - `deleteCard(id)`：optimistic 移除 → DELETE → 失敗回滾
  - `moveCard(id, toStatus, beforeId?, afterId?)`：optimistic 改欄 → POST move → 失敗回滾
- DragOverlay：拖拉時顯示卡片 ghost
- KeyboardSensor：方向鍵移動 + Enter 確認，符合 a11y

## 風險評估

- **`useKanbanBoard` 的 optimistic update 與 server 結果不同步**：失敗時必須完整回滾，否則 UI 會與 DB 漂移。對策：保留 `previousState` snapshot，失敗時直接 setState(previousState)
- **dnd-kit 的 sortable id 必須穩定**：用 `card.id`（cuid）作為 sortable id，**不可**用 index
- **拖拉「同欄、相同位置」邊界**：`handleDragEnd` 需檢查 `over.id === active.id`，直接 return 不送 API
- **session 沒 memberId**：Spec A 的 jwt callback 已塞入；若使用者 session 過期，API 回 401，前端要處理 redirect 至登入頁（沿用既有 middleware 行為）

### 目標確認

### 關鍵問答

#### sortOrder 為什麼用整數而非 LexoRank？

**AI 回應摘要**：面試規模卡片數 < 100，整數中位數演算法（間隔 1000，衝突時 normalize）足夠；LexoRank 引入第三方依賴與字串比較複雜度，過度工程。Plan 已採納此決策。

#### 為什麼把 Toaster 掛在 (dashboard)/layout 而非 root layout？

**AI 回應摘要**：root layout 是 Server Component，掛 Toaster 需 `'use client'` 標記造成全層 client 化。dashboard layout 已是 mixed，掛 Toaster 影響範圍小。登入頁不需要 toast（用既有的 inline error display）。

### 決策記錄

| 決策 | 結果 | 理由 |
| --- | --- | --- |
| Move API 採 `beforeId / afterId` 而非直接傳 sortOrder | ✅ 採納 | client 不需懂 sortOrder 算法；server 算術更可靠 |
| Card 回應不含 ownerId | ✅ 採納 | 防止 client 端被誤用以做跨使用者操作（後端守衛已存在，但前端不曝光更安全） |
| Service 層分離為獨立檔案 | ✅ 採納 | 業務邏輯（sortOrder、normalize）可單測；route 只剩薄轉發層 |
| 用 react-hot-toast 而非 RizzUI Toaster | ✅ 採納 | react-hot-toast 已在 package.json 且 roles/page.tsx 既已使用，避免引入新元件 |
| 主鍵 / 拖拉 ID 一律 `card.id`（cuid） | ✅ 採納 | dnd-kit 要求穩定 ID；用 cuid 已保證唯一 |
