## 1. 產品概述

在既有 Azeroth 面試作業骨架（Next.js 15 + Prisma + NextAuth + RBAC）上，新增一個**個人 Kanban 看板功能**，讓登入使用者能以四欄位（待處理 / 進行中 / 待驗收 / 已完成）管理自己的待辦事項，並透過直覺的拖拉操作更新卡片狀態。

本次任務同時把認證機制從既有的 NextAuth Credentials（帳密 + bcrypt）**切換為 Keycloak OIDC SSO**，以符合企業級單一登入的實務情境。

### 一句話定位

> 一個需登入、支援拖拉（跨欄＋欄內排序）、走 Keycloak SSO 的個人 Kanban 看板。

## 3. 使用者角色與場景

### 3.1 角色

本次任務僅保留三個語意清楚的角色：

| 角色 | 定位 | 典型使用者 |
| --- | --- | --- |
| **admin** | 系統管理員（兼具一般使用者身份） | 架設與維運者 |
| **user** | 一般使用者（只使用 Kanban） | 日常填寫待辦事項的人 |
| **viewer** | 唯讀訪客 | 被授權觀看但不可編輯的對象（例如 demo 帳號） |

### 3.2 權限矩陣（Role × Permission）

`✅` 表示具備此權限，`❌` 表示無。

| Permission Code | 說明 | admin | user | viewer |
| --- | --- | --- | --- | --- |
| `kanban.view` | 進入 Kanban 頁、檢視自己的卡片 | ✅ | ✅ | ✅ |
| `kanban.create` | 新增卡片 | ✅ | ✅ | ❌ |
| `kanban.edit` | 編輯卡片（含拖拉改狀態 / 排序） | ✅ | ✅ | ❌ |
| `kanban.delete` | 刪除卡片 | ✅ | ✅ | ❌ |
| `roles.view` | 檢視角色列表 | ✅ | ❌ | ❌ |
| `roles.create` | 新增角色 | ✅ | ❌ | ❌ |
| `roles.edit` | 編輯角色 | ✅ | ❌ | ❌ |
| `roles.delete` | 刪除角色 | ✅ | ❌ | ❌ |
| `user_roles.view` | 檢視使用者角色指派 | ✅ | ❌ | ❌ |
| `user_roles.edit` | 指派 / 調整使用者角色 | ✅ | ❌ | ❌ |
| `role_permissions.view` | 檢視 Role-Permission 指派 | ✅ | ❌ | ❌ |
| `role_permissions.edit` | 指派 / 調整 Role 持有的 permissions | ✅ | ❌ | ❌ |
| `audit_logs.view` | 檢視稽核紀錄 | ✅ | ❌ | ❌ |
| `login_records.view` | 檢視登入紀錄 | ✅ | ❌ | ❌ |

> **Ownership 原則**：每張 KanbanCard 都有 `ownerId`，使用者**只看得到且只能操作自己的卡片**，不論角色（admin 也不例外 — admin 看到的 Kanban 是 admin 自己的）。`kanban.edit / delete` 權限用於「是否具備此動作能力」的層級判斷；API 層另以 `where: { ownerId: session.userId }` 過濾確保不跨使用者。
>
> **viewer 的行為**：雖具 `kanban.view` 權限，但因 Ownership 過濾，viewer 在自己帳號下只看到自己的卡片（可能為空）；若需要 demo 預先填好卡片，請於該 viewer 帳號下手動建立。

### 3.3 關鍵場景

- **場景 A — 建立任務**
  使用者透過 Keycloak 登入 → 進入 `/admin/kanban` → 點「新增卡片」→ 於 Modal 輸入標題（必填）與描述（選填）→ 送出 → 卡片即時出現在「待處理」欄位。

- **場景 B — 更新進度（編輯）**
  點擊任一卡片 → Modal 開啟並帶入現值 → 修改描述並把狀態下拉改為「進行中」→ 儲存 → 卡片從原欄位移至新欄位。

- **場景 C — 拖拉推進（跨欄）**
  從「進行中」抓起卡片 → 拖至「待驗收」欄 → 釋放 → 卡片穩定停留於新欄，後端 `status` 同步更新為 `IN_REVIEW`。

- **場景 D — 欄內重排**
  於「待處理」欄把第 3 張卡片拖到第 1 位 → 釋放 → 該欄順序即時更新，後端 `sortOrder` 同步調整。

- **場景 E — 鍵盤操作**
  Tab 聚焦卡片 → Space 抓取 → 方向鍵移動 → Enter / Space 放下 → 後端同步更新（AC 4.5）。

- **場景 F — 刪除**
  打開編輯 Modal → 點「刪除」→ 二次確認 → 卡片從看板消失並從 DB 移除。

## 5. 資料模型（概念層）

### 5.1 KanbanCard（新）

| 屬性 | 型別 | 必填 | 備註 |
| --- | --- | --- | --- |
| `id` | 識別碼（UUID / cuid） | ✓ | 系統產生 |
| `title` | 字串（≤ 120 字） | ✓ | 卡片標題 |
| `description` | 字串（≤ 2000 字） | — | 卡片描述（可選） |
| `status` | Enum `CardStatus` | ✓ | `TODO` / `IN_PROGRESS` / `IN_REVIEW` / `DONE` |
| `sortOrder` | 整數 | ✓ | 欄內排序（同 status 下值越小越靠前；建議以浮點或間隔為 1000 的整數，便於中插） |
| `ownerId` | 識別碼（Member.id） | ✓ | 卡片擁有者；所有查詢以此過濾 |
| `createdAt` | 時戳 | ✓ | 系統產生 |
| `updatedAt` | 時戳 | ✓ | 系統產生 |

索引建議：`(ownerId, status, sortOrder)` 複合索引（看板載入主查詢）

### 5.2 Member（調整）

| 屬性 | 異動 | 備註 |
| --- | --- | --- |
| `password` | **改為 optional** 或移除 | 走 Keycloak 後應用端不再存密碼 |
| `keycloakSub` | **新增** | 對應 Keycloak 使用者的 `sub`，建 unique index；首次登入時 upsert |

### 5.3 Permission（新增 6 筆種子資料）

| code | name | description |
| --- | --- | --- |
| `kanban.view` | 檢視 Kanban | 可進入看板並檢視自己的卡片 |
| `kanban.create` | 新增卡片 | 可建立新卡片 |
| `kanban.edit` | 編輯卡片 | 可編輯卡片（含拖拉改狀態與排序） |
| `kanban.delete` | 刪除卡片 | 可刪除卡片 |
| `role_permissions.view` | 檢視 Role-Permission | 可檢視 Role 持有的 permissions |

## 10. Keycloak 整合與自動化啟動（§ Q1 / Q8 / Q9 延伸 + US-8）

### 10.1 架構影響

| 影響面 | 異動 |
| --- | --- |
| 認證機制 | NextAuth v5 Credentials Provider → **Keycloak Provider（OIDC）** |
| 部署 | `docker-compose.yml` 新增 `keycloak` service，與既有 `postgres` / `admin` 並列；加入 `depends_on` 與 `healthcheck` 串起啟動相依順序 |
| 環境變數 | 新增 `KEYCLOAK_ISSUER`、`KEYCLOAK_CLIENT_ID`、`KEYCLOAK_CLIENT_SECRET`、`NEXTAUTH_SECRET`、`NEXTAUTH_URL` |
| Prisma schema | `Member.password` 改為 optional；新增 `Member.keycloakSub`（unique） |
| 登入頁 | 從帳密表單改為「以 Keycloak 登入」按鈕（跳轉至 Keycloak） |
| 首次登入 | NextAuth `signIn` / `jwt` callback：根據 Keycloak `sub` 在 Member 表 upsert；根據 `realm_access.roles` 指派 Role |
| 登出 | OIDC RP-initiated logout（同時登出 Keycloak session） |
| Realm 初始化 | `keycloak/realm-export.json` 於 docker-compose 啟動時自動匯入 |

### 10.2 Keycloak 預設配置

- **Realm**：`kanban`
- **Client**：`kanban-admin`（confidential，啟用 standard flow；redirect URI `http://localhost:3010/api/auth/callback/keycloak`）
- **Realm roles**：`admin` / `user` / `viewer`

### 10.3 預設使用者（兩層）

**Layer 1 — Keycloak Realm 內建使用者**（從 `realm-export.json` 匯入）

| Email | 密碼（首次） | Keycloak Realm Role |
| --- | --- | --- |
| admin@example.com | Admin@1234 | admin |
| user@example.com | User@1234 | user |
| viewer@example.com | Viewer@1234 | viewer |

**Layer 2 — PostgreSQL Prisma Seed（`prisma/seed.ts`）**

- **Role 表**：`admin` / `user` / `viewer` 三筆
- **Permission 表**（共 **14** 筆）：
  - 既有：`roles.view/create/edit/delete`、`user_roles.view/edit`、`audit_logs.view`、`login_records.view`（8 筆）
  - 新增（v0.4）：`kanban.view/create/edit/delete`、`role_permissions.view/edit`（6 筆）
- **RolePermission 表**（對應 § 3.2 權限矩陣）：
  - `admin` → 全部 **14** 個 permissions
  - `user` → `kanban.view`、`kanban.create`、`kanban.edit`、`kanban.delete`（僅 Kanban 相關，不含後台管理）
  - `viewer` → 僅 `kanban.view`
- **Member 表**：**不 seed 帳號**（SSO 首次登入時由 NextAuth signIn callback 自動 upsert，以 keycloakSub 為鍵）

> 理由：走 OIDC 後 Member 表不再儲存密碼，密碼管理移交 Keycloak；Prisma seed 只保證「角色 / 權限骨架」存在，Member 由 SSO 第一次登入時 lazy create。

### 10.4 docker-compose 啟動與自動初始化流程

**服務組成與相依性**（從下到上啟動）：

```
┌─────────────────────────────────────────────────┐
│ admin (Next.js)                                 │
│   depends_on: postgres(healthy), keycloak(healthy)
│   entrypoint: prisma migrate deploy             │
│             → prisma db seed                    │
│             → next start                        │
└─────────────────────────────────────────────────┘
            ↓                    ↓
┌──────────────────┐   ┌──────────────────────────┐
│ postgres         │   │ keycloak                 │
│   healthcheck:   │   │   command: start-dev     │
│     pg_isready   │   │     --import-realm       │
│   volumes:       │   │   volumes:               │
│     pgdata       │   │     ./keycloak/realm-    │
│                  │   │       export.json →      │
│                  │   │     /opt/keycloak/data/  │
│                  │   │       import/            │
│                  │   │   healthcheck:           │
│                  │   │     curl /health/ready   │
└──────────────────┘   └──────────────────────────┘
```

**初始化步驟**（`docker compose up -d` 後自動執行）：

1. **postgres** 啟動 → 建立 `kanban` 資料庫（by env `POSTGRES_DB`）→ `pg_isready` healthcheck 通過
2. **keycloak** 啟動（依賴 postgres 健康）→ 偵測 `/opt/keycloak/data/import/` 內的 `realm-export.json` → 自動匯入 `kanban` realm（含 client、roles、三組預設使用者）→ healthcheck 通過
3. **admin** 啟動（依賴 postgres + keycloak 健康）→ 以 entrypoint 腳本（或 init container）依序執行：
   - `npx prisma migrate deploy`（套用 migration，若已最新則 no-op）
   - `npx prisma db seed`（`seed.ts` 內以 `upsert` 保證冪等，已存在不重建）
   - `next start`（啟動 Next.js）

**冪等保證**（AC 8.3）：
- Prisma seed 使用 `prisma.role.upsert` / `prisma.permission.upsert` / `prisma.rolePermission.upsert`，重啟不覆蓋
- Keycloak `--import-realm` 於 realm 已存在時跳過（或以 `IF_RESOURCE_MISSING` 策略），不覆蓋現有使用者密碼

**還原測試**（AC 8.4）：`docker compose down -v` 清除所有 volumes（`pgdata`、`keycloak-data`）後再 `up`，即可重現初始狀態。

**環境變數來源**：所有服務的 env（含 Keycloak admin 帳號、Postgres 帳密、NextAuth secret）統一放在 `.env`（本地）或 `.env.example`（範本），避免散落。

### 12.2 雙碼制 — `ApiReturnCode` + `ApiErrorCode`

依 `.claude/rules/coding-standards.md` 命名規範：型別 / 介面 PascalCase、常數值 UPPER_CASE、API JSON 欄位 camelCase。

```typescript
// common/src/api-response.ts（既有，保留）
export enum ApiReturnCode {
  SUCCESS = 0,
  VALIDATION_ERROR = 400,
  UNAUTHORIZED = 401,
  FORBIDDEN = 403,
  NOT_FOUND = 404,
  RATE_LIMITED = 429,
  INTERNAL_ERROR = 500,
}

// common/src/api-error-code.ts（v0.4 新增）
export const ApiErrorCode = {
  // Kanban
  KANBAN_CARD_NOT_FOUND: 'KANBAN_CARD_NOT_FOUND',
  KANBAN_CARD_TITLE_REQUIRED: 'KANBAN_CARD_TITLE_REQUIRED',
  KANBAN_CARD_TITLE_TOO_LONG: 'KANBAN_CARD_TITLE_TOO_LONG',
  KANBAN_CARD_DESCRIPTION_TOO_LONG: 'KANBAN_CARD_DESCRIPTION_TOO_LONG',
  KANBAN_CARD_FORBIDDEN: 'KANBAN_CARD_FORBIDDEN',
  KANBAN_INVALID_STATUS: 'KANBAN_INVALID_STATUS',
  // Role-Permission
  ROLE_NOT_FOUND: 'ROLE_NOT_FOUND',
  PERMISSION_NOT_FOUND: 'PERMISSION_NOT_FOUND',
  ROLE_PERMISSION_FORBIDDEN: 'ROLE_PERMISSION_FORBIDDEN',
  // 共用
  SESSION_EXPIRED: 'SESSION_EXPIRED',
  INTERNAL_ERROR: 'INTERNAL_ERROR',
} as const;
export type ApiErrorCode = typeof ApiErrorCode[keyof typeof ApiErrorCode];

// ApiResult<T> 擴充
export interface ApiResult<T> {
  success: boolean;
  code: ApiReturnCode;        // HTTP 級粗分類（既有）
  errorCode?: ApiErrorCode;   // 細粒度錯誤碼（新增；camelCase 欄位 / PascalCase 型別 / UPPER_CASE 值）
  message: string;
  data?: T;
  timestamp: number;
}
```

| 欄位 | 命名 | 用途 |
| --- | --- | --- |
| `code: ApiReturnCode` | 既有 | HTTP 級分類，driver 給通用 UI 行為（401 → 跳登入、403 → 顯示無權限、其他 → 通用錯誤） |

| `message: string` | 既有 | 中文預設訊息，server log 與 fallback 用 |

| `chip` | 登入頁 `/[locale]/admin/login`（pre-auth） | 圓形 / 膠囊狀 chip，置於登入卡片**右上角內側**（`absolute top-4 right-4`），label 顯示**目標語言**（當前 zh-TW 時 chip 顯示「EN」），點擊雙語循環切換 |

切換邏輯：將當前路徑 `[locale]` 段替換並寫入 cookie；下次無 locale 訪問時依 cookie 套用。

### 12.5 既有 Service 層回填策略

- 本次**不**大規模回頭改既有 service 的 errorCode
- **強制使用 `ApiErrorCode`** 的範圍：
  - Spec B — Kanban 所有 API（`/api/v1/kanban/cards/*`）
  - Spec A — 新增的 `RolePermission` 編輯 API
- 既有 API 保持 `errorCode = undefined`，前端 helper 遇 undefined 時直接顯示 `ApiResult.message`（向後相容）

- 三欄位：標題、描述（textarea）、狀態下拉（4 選項 + emoji prefix）

### 13.7 Toast

- 位置：viewport 右上 fixed（`fixed top-4 right-4`）
- 樣式：圓角 banner、漸層背景
  - 成功：綠色（`bg-green-500` 或漸層）
  - 失敗：紅色（`bg-red-500`）
- 內容：emoji + 訊息文字（從 `messages/{locale}.json` 取）
- 行為：fade-in 顯示，3 秒後 fade-out 自動消失；可堆疊（多個 toast 垂直排列）

### 13.8 Emoji 與 i18n 的關係

- emoji 為**跨語言共用**（zh-TW / en 都顯示 📋 / 🚀 / 👀 / ✅）
- 在 `messages/{locale}.json` 中將 emoji 直接寫入翻譯字串（e.g., `"TODO": "📋 待處理"` / `"TODO": "📋 To Do"`）
